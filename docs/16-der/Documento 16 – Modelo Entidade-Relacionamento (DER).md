1 – Introdução
1.1 Objetivo
Este documento apresenta o Modelo Entidade-Relacionamento (DER) do sistema Gestor de Territórios e Publicações (GTP).
Seu objetivo é definir a estrutura lógica dos dados que compõem o sistema, descrevendo as entidades de negócio, seus atributos, relacionamentos, cardinalidades, restrições de integridade e diretrizes para implementação no banco de dados PostgreSQL.
O modelo descrito neste documento constitui a base para a construção da camada de persistência da aplicação, orientando a implementação das entidades JPA/Hibernate, dos scripts de migração do Flyway, dos repositórios Spring Data JPA e dos serviços responsáveis pela manipulação das informações.
Além disso, o DER garante consistência entre os requisitos funcionais definidos nos documentos anteriores e a estrutura física do banco de dados.

1.2 Objetivos Específicos
Este documento busca:
•	identificar todas as entidades do domínio;
•	definir atributos e chaves primárias;
•	estabelecer relacionamentos entre entidades;
•	especificar cardinalidades;
•	definir restrições de integridade;
•	orientar a implementação das entidades JPA;
•	servir de referência para o banco PostgreSQL;
•	apoiar a geração dos diagramas de banco de dados;
•	facilitar manutenção e evolução do modelo.

1.3 Escopo
O Modelo Entidade-Relacionamento contempla todos os módulos do GTP, incluindo:
•	autenticação;
•	usuários;
•	perfis e permissões;
•	congregações;
•	territórios;
•	quadras;
•	endereços;
•	publicadores;

•	estudos bíblicos;
•	publicações;
•	notificações;
•	auditoria;
•	configurações;
•	relatórios;
•	indicadores.
Cada módulo será representado por um conjunto de entidades relacionadas entre si.

1.4 Objetivos do Modelo
O modelo deverá atender aos seguintes requisitos:
•	eliminar redundâncias desnecessárias;
•	garantir integridade referencial;
•	favorecer consultas eficientes;
•	suportar crescimento da base de dados;
•	permitir evolução do domínio sem impactos significativos;
•	facilitar implementação utilizando JPA/Hibernate.

1.5 Princípios de Modelagem
A modelagem do banco seguirá os seguintes princípios:
Normalização
As entidades deverão estar normalizadas até, no mínimo, a Terceira Forma Normal (3FN).
Quando necessário, poderão ser aplicadas técnicas controladas de desnormalização visando desempenho, desde que documentadas.

Integridade Referencial
Todos os relacionamentos utilizarão chaves estrangeiras devidamente definidas.
Não serão permitidos registros órfãos em entidades críticas.
Integridade de Domínio
Os dados armazenados deverão respeitar:
•	regras de negócio;
•	tipos de dados;
•	obrigatoriedade dos campos;
•	restrições de unicidade;
•	validações de domínio.

Independência da Aplicação
O modelo de dados deverá representar o domínio do negócio e não detalhes específicos da implementação da aplicação.

Evolução Controlada
Novas entidades poderão ser adicionadas futuramente preservando a compatibilidade com o modelo existente.
Todas as alterações estruturais deverão ser realizadas por meio de migrações versionadas utilizando Flyway.

2 – Diretrizes Gerais da Modelagem
2.1 Objetivo
Este capítulo define os princípios e padrões utilizados na construção do Modelo Entidade-Relacionamento (DER) do GTP.
As diretrizes aqui apresentadas têm como objetivos:
•	padronizar a modelagem de dados;
•	garantir integridade referencial;
•	facilitar a implementação com JPA/Hibernate;
•	promover desempenho e escalabilidade;
•	simplificar a manutenção e evolução do banco de dados.
Todas as entidades do sistema deverão respeitar essas diretrizes.

2.2 Arquitetura Lógica do Banco de Dados
O banco de dados do GTP será estruturado em torno dos principais domínios do negócio, organizados de forma modular.
Domínios principais
•	Segurança e Autenticação
•	Administração
•	Congregações
•	Territórios
•	Publicadores
•	Designações
•	Visitas
•	Estudos Bíblicos
•	Publicações
•	Configurações
•	Relatórios
•	Auditoria
Cada domínio agrupa entidades relacionadas funcionalmente, favorecendo a organização do modelo e a separação de responsabilidades.

2.3 Normalização
O modelo deverá seguir, como regra geral, a Terceira Forma Normal (3FN).
Os objetivos da normalização são:
•	eliminar redundâncias;
•	evitar inconsistências;
•	reduzir anomalias de atualização;
•	facilitar a manutenção.
Em situações específicas, poderá ser adotada desnormalização controlada para otimização de desempenho, desde que tecnicamente justificada e documentada.

2.4 Estratégia de Identificação das Entidades
Todas as entidades possuirão uma chave primária denominada:
id
Características:
•	tipo BIGINT;
•	geração automática pelo PostgreSQL;
•	imutável;
•	utilizada como identificador técnico da entidade.
Exemplo SQL:
id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY

2.5 Chaves Naturais e Unicidade
Além da chave primária, algumas entidades possuirão identificadores naturais que deverão ser únicos.
Exemplos:
Entidade	Campo
Usuário	e-mail
Congregação	código
Território	número
Publicador	matrícula (quando aplicável)
Esses campos deverão possuir restrições UNIQUE.

2.6 Relacionamentos
O modelo utilizará três tipos principais de relacionamentos.
Um para Um (1:1)
Aplicado quando duas entidades possuem dependência exclusiva.
Exemplo:
•	Usuário ↔ Preferências de Usuário.

Um para Muitos (1:N)
Relacionamento predominante no sistema.
Exemplos:
•	Congregação → Publicadores
•	Congregação → Territórios
•	Território → Endereços
•	Publicador → Designações

Muitos para Muitos (N:N)
Quando necessário, será implementado por meio de entidades associativas.
Exemplo:
Publicador ↔ Grupo de Serviço
Representação:
Publicador
     │
     ▼
Publicador_Grupo
     ▲
     │
   Grupo
Essa abordagem facilita a inclusão de atributos adicionais na relação, como data de ingresso, função ou status.

2.7 Cardinalidades
As cardinalidades deverão ser explicitamente definidas para todos os relacionamentos.
Exemplo:
Relacionamento	Cardinalidade
Congregação → Publicador	1:N
Congregação → Território	1:N
Território → Endereço	1:N
Publicador → Designação	1:N
Usuário → Perfil	N:1
Essas definições orientarão tanto o DER quanto o mapeamento JPA.

2.8 Integridade Referencial
Todos os relacionamentos deverão ser protegidos por chaves estrangeiras (FOREIGN KEY).
Regras gerais:
•	não permitir referências inexistentes;
•	impedir registros órfãos;
•	garantir consistência entre tabelas.
As políticas de exclusão (ON DELETE) e atualização (ON UPDATE) deverão ser definidas conforme a regra de negócio de cada relacionamento.

2.9 Campos Comuns
Grande parte das entidades compartilhará um conjunto padronizado de campos.
Campo	Finalidade
id	Identificador único
ativo	Indica se o registro está ativo
data_criacao	Data de criação
data_atualizacao	Última alteração
criado_por	Usuário responsável pela criação
atualizado_por	Usuário responsável pela alteração
versao	Controle de concorrência otimista
Esses campos facilitam auditoria, rastreabilidade e controle de alterações.

2.10 Auditoria
As entidades críticas deverão registrar informações de auditoria.
Campos mínimos:
•	data de criação;
•	usuário criador;
•	data da última alteração;
•	usuário responsável pela alteração.
Além disso, operações relevantes serão registradas em tabela específica de auditoria, conforme definido no Documento 08 – Segurança.

2.11 Soft Delete
Sempre que possível, será adotada a estratégia de exclusão lógica (soft delete).
Em vez de remover fisicamente um registro, será utilizado um indicador de inativação.
Exemplo:
Campo	Valor
ativo	false
Vantagens:
•	preservação do histórico;
•	recuperação de registros;
•	manutenção da integridade referencial;
•	suporte à auditoria.
A exclusão física será reservada para situações específicas, como dados temporários ou exigências legais.

2.12 Controle de Concorrência
Será adotado o mecanismo de Concorrência Otimista utilizando um campo de versão.

Exemplo:
versao
Esse campo será incrementado automaticamente a cada atualização, evitando sobrescritas simultâneas.
No JPA, será implementado por meio da anotação @Version.

2.13 Convenções para Tipos de Dados

2.14 Índices

A criação excessiva de índices deverá ser evitada para não comprometer operações de escrita.

4 – Relacionamentos e Cardinalidades
4.1 Objetivo
Este capítulo define os relacionamentos existentes entre as entidades do GTP, especificando:
•	tipo do relacionamento;
•	cardinalidade;
•	entidade proprietária da relação;
•	chave estrangeira correspondente;
•	regras de integridade;
•	comportamento esperado durante operações de inserção, atualização e exclusão.
Essas definições garantem consistência entre o modelo conceitual, o banco de dados PostgreSQL e a implementação em JPA/Hibernate.

4.2 Tipos de Relacionamento
O modelo utilizará quatro tipos fundamentais.
Um para Um (1:1)
Cada registro de uma entidade está associado a um único registro da outra entidade.
Exemplos:
•	Usuário ↔ Preferências do Usuário
•	Usuário ↔ Configuração Individual (caso implementada)
Representação
Usuario
   │1
   │
   │1
PreferenciaUsuario
Implementação JPA
@OneToOne
@JoinColumn(name = "usuario_id")

5 – Integridade Referencial e Regras de Banco
5.1 Objetivo
Este capítulo estabelece as regras de integridade aplicáveis ao banco de dados do GTP.
São definidos:
•	tipos de restrições;
•	regras de integridade referencial;
•	políticas de atualização e exclusão;
•	validações de domínio;
•	mecanismos de auditoria;
•	controle de concorrência;
•	diretrizes para evolução do esquema de banco.
Essas definições asseguram que os dados permaneçam consistentes durante todo o ciclo de vida da aplicação.

5.2 Integridade de Entidade
Toda tabela deverá possuir uma chave primária (PRIMARY KEY) responsável por identificar unicamente cada registro.
Diretrizes
•	nome padronizado: id;
•	tipo BIGINT;
•	geração automática pelo PostgreSQL utilizando GENERATED BY DEFAULT AS IDENTITY;
•	valor imutável após a criação.
Exemplo:
CREATE TABLE usuario (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
);

5.3 Integridade Referencial
Os relacionamentos entre entidades deverão ser protegidos por chaves estrangeiras (FOREIGN KEY), impedindo referências inválidas.
Objetivos
•	evitar registros órfãos;
•	garantir consistência entre tabelas;
•	preservar o histórico de informações;
•	impedir exclusões que comprometam o domínio.
Exemplo:
ALTER TABLE publicador
ADD CONSTRAINT fk_publicador_congregacao
FOREIGN KEY (congregacao_id)
REFERENCES congregacao(id);

5.4 Restrições de Unicidade
Campos que representam identificadores naturais deverão possuir restrições UNIQUE.
Exemplos
Entidade	Campo
Usuário	email
Congregação	codigo
Território	numero
Perfil	nome
Permissão	codigo
Essas restrições evitam duplicidade de registros e reforçam a integridade do domínio.

5.5 Restrições CHECK
Sempre que possível, serão utilizadas restrições CHECK para validar valores diretamente no banco.
Exemplos
•	status válidos;
•	valores monetários não negativos;
•	datas coerentes;
•	campos numéricos dentro de limites definidos.
Exemplo:
CHECK (quantidade >= 0)
Outro exemplo:
CHECK (status IN ('ATIVO','INATIVO'))
Essas validações complementam as verificações realizadas pela aplicação.

5.6 Política de Exclusão
A exclusão física de registros será evitada sempre que possível.
Será adotada preferencialmente a estratégia de soft delete.
Campo padrão
ativo BOOLEAN
Valores:
Valor	Significado
true	Registro ativo
false	Registro inativo
Vantagens
•	preservação do histórico;
•	recuperação de registros;
•	suporte à auditoria;
•	manutenção da integridade referencial.
A exclusão física será reservada para dados temporários, registros auxiliares ou situações previstas em requisitos legais.

5.7 Políticas ON DELETE e ON UPDATE
As ações associadas às chaves estrangeiras seguirão critérios definidos conforme o domínio.
Situação	Política Recomendada
Entidades críticas	ON DELETE RESTRICT
Tabelas associativas	ON DELETE CASCADE
Relacionamentos opcionais	ON DELETE SET NULL
Alteração de chaves	ON UPDATE CASCADE (quando aplicável)
Exemplos
RESTRICT
Impedir exclusão de uma congregação que ainda possua publicadores vinculados.
CASCADE
Excluir automaticamente registros de uma tabela associativa quando o registro principal for removido.
SET NULL
Remover a referência quando o relacionamento for opcional, preservando o registro.

5.8 Integridade de Domínio
Os valores armazenados deverão respeitar as regras de negócio definidas no Documento 03.
Exemplos:
•	um território deve pertencer a uma congregação;
•	um publicador ativo deve estar vinculado a uma congregação;
•	uma designação deve referenciar um território existente;
•	um estudo bíblico deve possuir um publicador responsável.
Essas regras serão implementadas tanto na aplicação quanto, sempre que possível, no banco de dados.

5.9 Integridade Temporal
Campos de data deverão seguir regras cronológicas.
Exemplos:
•	data de devolução não pode ser anterior à data de designação;
•	data de atualização não pode ser anterior à data de criação;
•	data de expiração de um token deve ser posterior à data de emissão.
Essas validações podem ser reforçadas por CHECK, quando aplicável, ou pela camada de aplicação.


5.10 Auditoria dos Dados

Campos comuns:
Campo	Finalidade

5.11 Controle de Concorrência
Será adotada a estratégia de concorrência otimista (optimistic locking).
Cada entidade persistente possuirá o campo:
versao BIGINT
Na implementação JPA, esse campo será anotado com @Version.
Benefícios:
•	evita perda de atualizações;
•	detecta conflitos entre usuários;
•	melhora a escalabilidade em relação ao bloqueio pessimista.

6.2 Estratégia Geral de Indexação
A criação de índices deverá priorizar colunas utilizadas com frequência em:

•	ordenações (ORDER BY);
•	agrupamentos (GROUP BY);
•	junções (JOIN);
•	filtros compostos;
•	chaves estrangeiras.
Os índices deverão ser planejados de forma equilibrada, considerando que também impactam operações de escrita (INSERT, UPDATE e DELETE).

6.3 Índices Primários
Todas as tabelas possuirão índice automático sobre a chave primária.
Exemplo:
PRIMARY KEY (id)
Características:
•	único;
•	criado automaticamente pelo PostgreSQL;
•	utilizado em pesquisas por identificador.

designacao	publicador_id
designacao	territorio_id

Índice	Utilização

A ordem das colunas deverá refletir os padrões reais de consulta da aplicação.
6.7 Índices Parciais
Quando apenas parte dos registros é consultada com frequência, poderão ser utilizados índices parciais.
Exemplo:
CREATE INDEX idx_publicador_ativo
ON publicador (congregacao_id)
WHERE ativo = true;
Vantagens:
•	menor consumo de armazenamento;
•	melhor desempenho em consultas específicas;
•	redução do custo de manutenção do índice.

6.8 Consultas Otimizadas
As consultas deverão:
•	selecionar apenas as colunas necessárias;
•	utilizar filtros indexados;
•	evitar operações desnecessárias;
•	limitar resultados quando apropriado.
Exemplo recomendado:
SELECT id, nome
FROM publicador
WHERE congregacao_id = ?
ORDER BY nome
LIMIT 20;
Evitar:
SELECT *
FROM publicador;
quando nem todas as colunas forem necessárias.



6.9 Paginação
Grandes volumes de dados deverão ser paginados.
No PostgreSQL:
LIMIT 20 OFFSET 40;
Na API REST:
GET /publicadores?page=2&size=20
Na camada JPA:
•	Pageable;
•	Page<T>;
•	Slice<T> para consultas sem necessidade de contagem total.
A paginação reduz o consumo de memória e melhora o tempo de resposta.

6.10 Otimização de Relacionamentos
Para evitar consultas excessivas (N+1 Query Problem), deverão ser adotadas estratégias de carregamento adequadas.
Recomendações
•	utilizar LAZY como padrão para coleções;
•	aplicar JOIN FETCH quando necessário;
•	utilizar projeções (DTOs) para consultas específicas;
•	evitar EAGER indiscriminadamente.
Essas práticas reduzem a quantidade de consultas executadas e melhoram o desempenho geral.

6.11 Análise de Planos de Execução
Consultas críticas deverão ser analisadas periodicamente utilizando:
EXPLAIN
e
EXPLAIN ANALYZE
Essas ferramentas permitem identificar:
•	uso inadequado de índices;
•	varreduras completas (Sequential Scan);
•	junções ineficientes;
•	gargalos de desempenho.
Os resultados deverão orientar ajustes na modelagem e na indexação.

6.12 Estatísticas e Manutenção
O PostgreSQL utiliza estatísticas para otimizar a execução das consultas.
Recomenda-se a execução periódica de:

e

8 – Mapeamento para JPA/Hibernate
8.1 Objetivo
Este capítulo define as diretrizes para o mapeamento das entidades do Modelo Entidade-Relacionamento (DER) para objetos Java utilizando Jakarta Persistence (JPA) e Hibernate.
São abordados:
•	organização das entidades;
•	herança e reutilização;
•	mapeamento de atributos;
•	relacionamentos;
•	estratégias de carregamento;
•	cascatas;
•	auditoria;
•	concorrência otimista;
•	boas práticas para Spring Data JPA.
O objetivo é assegurar que a camada de persistência reflita fielmente o modelo de domínio e mantenha alto desempenho, integridade e facilidade de manutenção.

8.2 Estrutura do Pacote de Entidades
As entidades serão organizadas por domínio funcional, acompanhando a arquitetura modular do projeto.
src/main/java

    ├── assignment
    │   └── entity
    ├── visit
    │   └── entity

    │   └── entity
    ├── notification
    │   └── entity

    │   └── entity
    └── common
        └── Entity

Essa organização facilita a manutenção e reduz o acoplamento entre módulos.

8.3 Classe Base (BaseEntity)
As entidades compartilharão uma superclasse abstrata contendo atributos comuns.
Campos compartilhados
Campo	Finalidade
id	Identificador único
ativo	Exclusão lógica
dataCriacao	Auditoria
dataAtualizacao	Auditoria
criadoPor	Usuário criador
atualizadoPor	Usuário responsável pela alteração
versao	Controle de concorrência otimista
A classe será anotada com:
•	@MappedSuperclass
•	@EntityListeners
•	@Version (campo versao)
Essa abordagem evita duplicação de código e padroniza a persistência.

8.4 Identificadores
Todas as entidades utilizarão:
•	@Id;
•	@GeneratedValue(strategy = GenerationType.IDENTITY).
Exemplo conceitual:
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;
Essa estratégia é compatível com o PostgreSQL e simplifica a geração de chaves primárias.

8.5 Mapeamento de Atributos
Os atributos serão mapeados utilizando @Column.
Diretrizes
•	nomes em snake_case;
•	comprimento definido para campos VARCHAR;
•	obrigatoriedade refletida por nullable = false;
•	unicidade representada por unique = true, quando aplicável.
Exemplo:
@Column(name = "email", nullable = false, unique = true, length = 150)
private String email;

8.6 Relacionamentos
Os relacionamentos seguirão diretamente o DER definido no Capítulo 7.
Um para Um
Utilizar:
•	@OneToOne
•	@JoinColumn
Um para Muitos
Utilizar:
•	@OneToMany(mappedBy = "...")
Muitos para Um
Utilizar:
•	@ManyToOne
•	@JoinColumn
Muitos para Muitos
Preferencialmente implementado por entidades associativas.
Quando necessário:
•	@ManyToMany
•	@JoinTable
Essa abordagem proporciona maior flexibilidade para evolução do domínio.

8.7 Estratégias de Carregamento
Para otimizar o desempenho, serão adotadas estratégias adequadas de carregamento das associações.

LAZY
Será o padrão para coleções e relacionamentos que não precisam ser carregados imediatamente.
Vantagens:
•	menor consumo de memória;
•	redução do número de consultas;
•	melhor desempenho em operações de listagem.
EAGER
Será utilizado apenas quando o carregamento imediato for essencial para a regra de negócio.
O uso indiscriminado de EAGER deverá ser evitado para prevenir o problema conhecido como N+1 Query.

8.8 Cascatas
As operações em entidades relacionadas deverão utilizar CascadeType apenas quando houver dependência de ciclo de vida.
Principais estratégias:
Tipo	Utilização
PERSIST	Persistência conjunta
MERGE	Atualização conjunta
REMOVE	Exclusão de dependentes
REFRESH	Atualização dos dados persistidos
DETACH	Remoção do contexto de persistência
ALL	Utilizado apenas quando apropriado
O uso de CascadeType.ALL será restrito a entidades fortemente dependentes.

8.9 Exclusão Lógica (Soft Delete)
As entidades de negócio utilizarão exclusão lógica.
Campo padrão:
ativo BOOLEAN
A aplicação deverá considerar apenas registros ativos nas consultas padrão.
A exclusão física será reservada para dados temporários ou requisitos específicos.

8.10 Auditoria
As entidades deverão registrar automaticamente:
•	data de criação;
•	usuário criador;
•	data de atualização;
•	usuário responsável pela alteração.
A auditoria será integrada ao Spring Data JPA por meio de AuditingEntityListener.
Essa abordagem elimina a necessidade de preenchimento manual desses campos.

8.11 Controle de Concorrência
Será adotada concorrência otimista.
Campo:
versao
Mapeado com:
@Version
private Long versao;
Benefícios:
•	prevenção de sobrescritas simultâneas;
•	detecção de conflitos de atualização;
•	maior escalabilidade em ambientes com múltiplos usuários.

8.12 Validações
As entidades utilizarão Jakarta Validation.
Principais anotações:
Anotação	Finalidade
@NotNull	Campo obrigatório
@NotBlank	Texto obrigatório
@Email	Validação de e-mail
@Size	Tamanho mínimo/máximo
@Pattern	Expressões regulares
@Positive	Valores positivos
@Past	Datas passadas
@Future	Datas futuras
Essas validações complementam as restrições do banco de dados.

8.13 Enumerações
Valores controlados serão representados por enum.
Exemplos:
•	StatusTerritorio
•	StatusDesignacao
•	TipoPublicacao
•	Sexo
•	TipoNotificacao
O armazenamento preferencial será por nome (EnumType.STRING), garantindo maior legibilidade e evitando problemas decorrentes da alteração da ordem dos valores.

8.14 Repositórios
Cada entidade persistente possuirá um repositório específico baseado em JpaRepository.
Exemplo de responsabilidades:
•	operações CRUD;
•	consultas por critérios;
•	paginação;
•	ordenação;
•	consultas personalizadas.
Consultas complexas deverão utilizar projeções, Specification ou QueryDSL (quando adotado).

8.15 Consultas e Performance
Para manter a eficiência da camada de persistência, deverão ser observadas as seguintes práticas:
•	utilizar paginação (Pageable) em listagens;
•	evitar carregamento desnecessário de relacionamentos;
•	utilizar JOIN FETCH quando necessário;
•	preferir DTOs para consultas específicas;
•	revisar consultas geradas automaticamente pelo Hibernate.

8.16 Transações
Operações que alteram o estado do banco deverão ser executadas dentro de transações.
Diretrizes:
•	utilizar @Transactional na camada de serviço;
•	manter transações curtas;
•	evitar chamadas externas dentro de transações;
•	garantir atomicidade das operações críticas.

8.17 Integração com Flyway
O mapeamento JPA deverá permanecer sincronizado com as migrações do Flyway.
Fluxo recomendado:
1.	atualização do DER;
2.	criação da migração Flyway;
3.	atualização das entidades JPA;
4.	implementação dos repositórios;
5.	validação em ambiente de desenvolvimento.
Essa sequência reduz inconsistências entre o código e o banco de dados.

9 – Evolução do Modelo e Migrações
9.1 Objetivo
Este capítulo define a estratégia para evolução do modelo de dados do Gestor de Territórios e Publicações (GTP), estabelecendo regras para criação, alteração e manutenção do esquema do banco de dados ao longo do ciclo de vida da aplicação.
São abordados:
•	versionamento do banco de dados;
•	utilização do Flyway;
•	evolução incremental do modelo;
•	compatibilidade entre versões;
•	governança das alterações estruturais;
•	boas práticas para migrações.
Essas diretrizes garantem que o banco de dados evolua de forma previsível, auditável e consistente entre todos os ambientes.

9.2 Princípios da Evolução do Modelo
A evolução do esquema do banco seguirá os seguintes princípios:
•	versionamento obrigatório de todas as alterações;
•	migrações incrementais e imutáveis;
•	compatibilidade entre ambientes;
•	rastreabilidade completa;
•	automatização da aplicação das migrações;
•	preservação da integridade dos dados.
Nenhuma alteração estrutural deverá ser realizada diretamente no banco de dados em produção.

9.3 Utilização do Flyway
O Flyway será a ferramenta oficial para controle de versão do esquema do banco de dados.
Suas responsabilidades incluem:
•	criação de tabelas;
•	inclusão de colunas;
•	alteração de restrições;
•	criação de índices;
•	criação de views;
•	atualização de dados estruturais;
•	controle do histórico de migrações.
Todas as migrações serão executadas automaticamente durante a inicialização da aplicação.

9.4 Estrutura das Migrações
Os scripts serão organizados em diretório específico.
src/
└── main/
    └── resources/
        └── db/
            └── migration/
                ├── V1__criacao_usuario.sql
                ├── V2__criacao_congregacao.sql
                ├── V3__criacao_territorio.sql
                ├── V4__criacao_publicador.sql
                ├── V5__relacionamentos.sql
                └── ...
Cada arquivo representará uma única alteração lógica no esquema do banco.

9.5 Convenção de Nomenclatura
As migrações seguirão o padrão:
V<versão>__<descrição>.sql
Exemplos:
V1__criacao_tabela_usuario.sql
V2__criacao_tabela_perfil.sql
V3__criacao_tabela_congregacao.sql
V4__criacao_relacionamentos.sql
V5__criacao_indices.sql
Diretrizes:
•	numeração sequencial;
•	descrição clara e objetiva;
•	uso de letras minúsculas e snake_case;
•	evitar abreviações pouco compreensíveis.

9.6 Versionamento Incremental
Cada modificação estrutural deverá resultar em uma nova versão do esquema.
Exemplos de alterações que exigem nova migração:
•	criação de tabela;
•	inclusão de coluna;
•	alteração de tipo de dado;
•	criação de índice;
•	inclusão de restrição;
•	alteração de relacionamento;
•	criação de view;
•	inserção de dados de referência.
Scripts já executados não deverão ser modificados.

9.7 Compatibilidade entre Versões
A evolução do banco deverá preservar a compatibilidade entre diferentes versões da aplicação sempre que possível.
Recomendações:
•	evitar remoção imediata de colunas utilizadas;
•	realizar alterações em etapas;
•	manter versões intermediárias compatíveis durante implantações graduais;
•	comunicar alterações que exijam adaptação da aplicação.
Essa estratégia reduz riscos durante atualizações em produção.

9.8 Migrações de Dados
Além de alterações estruturais, poderão existir migrações destinadas à transformação ou inicialização de dados.

Exemplos:
•	inclusão de perfis padrão;
•	cadastro de permissões iniciais;
•	parâmetros do sistema;
•	configurações globais;
•	dados de referência.
Essas migrações deverão ser idempotentes sempre que possível, evitando efeitos indesejados em reexecuções controladas.

9.9 Ambientes
As migrações deverão ser aplicadas de forma consistente em todos os ambientes.
Fluxo recomendado:
  Desenvolvimento
        │
        ▼
   Homologação
        │
        ▼
    Produção
A ordem e o conteúdo das migrações deverão permanecer idênticos entre os ambientes.

9.10 Controle de Qualidade
Antes da liberação de uma migração, recomenda-se validar:
•	sintaxe SQL;
•	impacto sobre dados existentes;
•	tempo estimado de execução;
•	compatibilidade com a versão atual da aplicação;
•	reversibilidade (quando aplicável);
•	impacto em índices e desempenho.
Essas verificações reduzem riscos durante implantações.


9.11 Estratégia para Alterações Complexas
Alterações de grande impacto deverão ser realizadas em etapas.
Fluxo recomendado:
1.	adicionar novas estruturas;
2.	migrar os dados;
3.	adaptar a aplicação;
4.	validar funcionamento;
5.	remover estruturas antigas em migração posterior.
Essa abordagem minimiza indisponibilidade e facilita a reversão de mudanças.

9.12 Governança das Alterações
Toda alteração estrutural deverá ser registrada e aprovada conforme o fluxo de desenvolvimento do projeto.
Cada migração deverá possuir:
